# 출판 원고 인수인계 노트

## 1. 이 문서의 목적

이 문서는 GPT 5.5, Claude Opus 4.8, Fable에게 출판 원고 집필을 넘길 때 함께 제공하는 최종 인수인계 문서다.

이 문서의 역할은 원고 전체의 방향을 다시 고정하는 것이다.

집필 모델은 여러 파일을 참고하게 된다. 이때 파일이 많아지면 전체 의도가 흐려질 수 있다. 따라서 이 문서에서는 “무엇을 반드시 지켜야 하는지”, “무엇을 더 확장해야 하는지”, “무엇을 조심해야 하는지”를 한 번 더 정리한다.

## 2. 이번 원고의 가장 중요한 전제

이번 프로젝트는 실제 코드를 수정하는 프로젝트가 아니다.

이번 단계는 출판 원고의 양과 품질을 높이는 단계다.

현재 구축된 `phoenix pages`는 V0 상태로 둔다. V1, V1.5, V2, V3는 앞으로의 확장 계획으로 설명한다.

집필 모델은 아래 경계를 지켜야 한다.

1. 현재 운영 중인 웹사이트 기능을 실제로 수정한 것처럼 쓰지 않는다.
2. 아직 만들지 않은 agent web을 이미 완성된 서비스처럼 쓰지 않는다.
3. 예정 저장소는 예정이라고 표시한다.
4. 실제 존재 저장소는 실제 존재한다고 표시한다.
5. Agent Bot v2.0은 웹컨트롤형 목표 버전으로 설명한다.

## 3. 반드시 유지할 버전 구분

원고에서 버전은 아래처럼 구분한다.

```text
phoenix pages V0 = 현재 사람이 직접 사용하는 기본 상세페이지 생성 웹
phoenix pages V1 = 로그인, Supabase Auth, Database, Storage 적용
phoenix pages V1.5 = 결제와 크레딧 적용
phoenix pages V2 = 봇 자동화 준비 구조 강화
Phoenix Agent v2.0 = Web Control Agent
Phoenix Agent v2.2 = Master Builder Agent
```

주의:

V2와 Agent v2.0은 같은 말이 아니다.

웹사이트 버전과 에이전트 봇 버전은 서로 연결되지만, 이름이 같다고 같은 시스템은 아니다.

집필 모델은 이 차이를 독자에게 쉽게 설명해야 한다.

## 4. 반드시 유지할 배포 기준

배포 기준은 아래와 같다.

```text
phoenix pages = Vercel
나머지 agent web = Netlify
```

이 기준은 원고 전체에서 흔들리면 안 된다.

단, 장시간 생성 작업이나 대용량 파일 처리는 플랫폼만으로 해결되지 않을 수 있다. 이때는 백엔드 구조, 비동기 작업, Supabase Storage, Cloud 링크, outputs 저장을 함께 고려해야 한다고 설명한다.

## 5. 반드시 유지할 봇 구조

봇 구조는 아래와 같다.

```text
genesis_bot = Genesis Bot
design_bot = Design Bot
writer_bot = Writer Bot
video_bot = Video Bot
power_bot = Power Bot
```

Genesis Bot은 모든 일을 직접 하지 않는다.

Genesis Bot은 일을 나누고, 전문 봇에게 넘기고, 상태를 추적하고, 최종 보고한다.

전문 봇은 자기 담당 agent web을 조작한다.

이 구조는 책의 중심 구조이므로, 집필 모델은 반드시 유지한다.

## 6. 원고를 더 늘릴 때 우선순위

원고를 추가로 늘릴 때는 아래 순서를 따른다.

1. 실제 독자가 따라 할 절차를 늘린다.
2. 실패했을 때 확인할 순서를 늘린다.
3. 봇별 대화 예시를 늘린다.
4. agent web별 실제 사용 장면을 늘린다.
5. 승인 정책과 보안 설명을 늘린다.
6. 공식 문서 기반 근거를 늘린다.

늘리면 안 되는 것:

1. 같은 뜻의 문장 반복
2. 추상적인 찬사
3. 검증되지 않은 기능 약속
4. 가격과 정책 단정
5. 위험 작업을 자동으로 진행하는 표현

## 7. 최종 합본 방식

여러 모델이 원고를 나누어 쓸 경우 아래 방식으로 합본한다.

1. GPT 5.5는 구조와 절차를 담당한다.
2. Claude Opus 4.8은 긴 설명과 개념 확장을 담당한다.
3. Fable은 문장 리듬과 사례 서술을 담당한다.
4. 최종 합본자는 안전 검수 체크리스트로 다시 검토한다.
5. 기술 근거가 필요한 문장은 검증 리서치 노트와 대조한다.
6. 중복 문단은 합치고, 서로 다른 표현은 더 쉬운 쪽을 남긴다.

## 8. 마지막 점검 문장

집필 모델은 원고를 마무리하기 전에 아래 질문에 답해야 한다.

1. 이 장은 입문자가 이해할 수 있는가?
2. 이 장에는 실제로 따라 할 단계가 있는가?
3. 이 장에는 실패했을 때 확인할 방법이 있는가?
4. 이 장은 보안과 승인 정책을 어기지 않는가?
5. 이 장은 Phoenix 시스템의 핵심 메시지와 연결되는가?

핵심 메시지는 다시 아래 문장이다.

```text
Agent Webs는 사람이 쓰는 화면이자, 봇이 일하는 작업장이다.
```

이 문장을 기준으로 원고 전체를 다시 정리하면, 책의 방향이 흐트러지지 않는다.
