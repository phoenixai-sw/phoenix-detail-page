# 집필 모델용 원고작성 브리프

## 1. 이 문서의 목적

이 문서는 GPT 5.5, Claude Opus 4.8, Fable 같은 집필 모델이 `age of agent - phoenix agent bots & agent webs` 원고를 확장할 때 기준으로 삼는 작성 브리프다.

집필 모델은 이 문서를 단순 요약하지 않는다. 이 문서를 기준으로 장별 원고를 풍부하게 확장하고, 초보자가 실제로 따라 할 수 있는 문장과 절차를 만든다.

핵심 목표는 세 가지다.

1. 독자가 agent bot과 agent web의 개념을 이해한다.
2. 독자가 GitHub, Vercel, Netlify, Telegram, Phoenix Agent v2.0 설치 흐름을 따라 한다.
3. 독자가 자동화의 편리함과 승인/보안/검증의 중요성을 함께 이해한다.

## 2. 전체 톤

문체는 입문자용 실전서 톤으로 쓴다.

너무 개발자 문서처럼 차갑게 쓰지 않는다. 반대로 홍보 문구처럼 과장하지도 않는다.

좋은 문장은 아래 조건을 만족한다.

1. 한 문단에는 하나의 핵심만 담는다.
2. 기술 용어가 나오면 바로 쉬운 설명을 붙인다.
3. 절차는 번호를 붙인다.
4. 위험한 작업은 먼저 멈추고 확인하도록 안내한다.
5. 예시는 실제 운영자가 말할 법한 문장으로 쓴다.
6. token, API key, OAuth credential 같은 민감값은 절대 예시 실값으로 쓰지 않는다.

## 3. 금지되는 원고 방식

집필 모델은 아래 방식으로 분량을 늘리면 안 된다.

1. 같은 문장을 표현만 바꿔 반복하지 않는다.
2. 의미 없는 수식어를 늘리지 않는다.
3. 검증되지 않은 모델명, 가격, 정책을 단정하지 않는다.
4. 세무/의료 결과를 확정 상담처럼 쓰지 않는다.
5. 배포 플랫폼의 제한을 무시하고 “무조건 된다”고 쓰지 않는다.
6. 봇이 사용자의 승인 없이 비용, 삭제, 배포를 진행한다고 쓰지 않는다.
7. 실제 secret 값이 들어간 예시를 만들지 않는다.

## 4. 반드시 유지할 핵심 메시지

이 책의 중심 문장은 아래와 같다.

```text
Agent Webs는 사람이 쓰는 화면이자, 봇이 일하는 작업장이다.
```

이 문장을 각 장에서 다른 방식으로 되살려야 한다.

1장에서는 시대 변화로 설명한다.

2장에서는 웹사이트의 역할 변화로 설명한다.

3장에서는 phoenix command의 지휘 구조로 설명한다.

실전 장에서는 data-testid, Playwright MCP, outputs 저장, Telegram 선택 전송으로 설명한다.

운영 장에서는 승인 정책과 실패 대응으로 설명한다.

## 5. 책의 독자상

독자는 개발 전문가가 아니다.

독자는 아래 수준으로 가정한다.

1. GitHub라는 말을 들어봤지만 clone이 익숙하지 않을 수 있다.
2. Vercel과 Netlify가 배포 플랫폼이라는 것은 알지만 설정 화면은 낯설 수 있다.
3. Telegram BotFather로 봇을 만든 경험이 없을 수 있다.
4. API key와 환경변수가 왜 중요한지 아직 감각이 부족할 수 있다.
5. 자동화를 좋아하지만, 자동화가 위험해지는 지점을 잘 모를 수 있다.

따라서 집필 모델은 “이미 아는 사람”이 아니라 “처음 따라 하는 사람”에게 말해야 한다.

## 6. 장별 집필 방향

### 1장: 에이전트 시대가 온다

이 장에서는 AI가 답변 도구에서 실행 도구로 바뀌는 흐름을 설명한다.

핵심은 “AI가 똑똑해졌다”가 아니다. 핵심은 “AI가 실제 업무 흐름 안으로 들어온다”이다.

예시는 이렇게 잡는다.

1. 예전에는 사람이 웹사이트에 접속했다.
2. 사람이 파일을 올렸다.
3. 사람이 버튼을 눌렀다.
4. 사람이 다운로드했다.
5. 이제는 봇이 이 절차 중 일부를 맡을 수 있다.

### 2장: agent webs란 무엇인가

이 장에서는 agent web을 정의한다.

일반 웹사이트는 사람에게 보이는 화면이다.

agent web은 사람에게 보이면서 동시에 봇에게도 읽히는 작업장이다.

차이를 설명할 때는 `data-testid`를 예로 들어도 좋다. 사람에게는 “결과 다운로드”라는 글자가 보이면 충분하지만, 봇에게는 안정적인 선택자가 필요하다.

### 3장: phoenix command

이 장에서는 지휘 웹이 왜 필요한지 설명한다.

agent web이 많아지면 각각의 작업 상태를 사람이 기억할 수 없다. 상세페이지는 어디까지 생성되었는지, 제품컷은 누가 맡았는지, PPT는 실패했는지, 결과물은 어디에 저장되었는지 흩어진다.

phoenix command는 이 흩어진 작업을 한 화면에 모으는 역할이다.

### 4장: Phoenix Agent Bots

이 장에서는 봇별 역할을 명확히 나눈다.

Genesis Bot은 모든 일을 직접 하지 않는다. Genesis Bot은 일을 나누고 배정한다.

Design Bot은 시각 결과물을 맡는다.

Writer Bot은 텍스트와 문서 결과물을 맡는다.

Video Bot은 영상 작업을 맡는다.

Power Bot은 리서치, 상담, 대시보드형 업무를 맡는다.

### 5장: Phoenix Agent Webs

이 장에서는 각 agent web을 하나씩 소개한다.

각 웹마다 아래 항목을 넣는다.

1. 무엇을 만드는 웹인가?
2. 누가 컨트롤하는가?
3. 결과물은 무엇인가?
4. 사람은 어떤 때 직접 쓰는가?
5. 봇은 어떤 때 대신 조작하는가?
6. 위험하거나 승인 필요한 지점은 무엇인가?

### 6장 이후: 실전 장

실전 장에서는 독자가 직접 따라 할 수 있어야 한다.

GitHub, Vercel, Netlify, Telegram, Agent v2.0 설치, outputs, 승인 정책, 실패 대응 순서로 쓴다.

각 단계는 “목표 - 준비물 - 절차 - 성공 기준 - 자주 막히는 지점 - 다음 단계” 구조로 쓰면 좋다.

## 7. 반드시 설명해야 할 운영 원칙

### v1.9 보존

Phoenix Agent v2.0은 v1.9 안정판을 깨지 않는다.

집필 모델은 이 원칙을 반복해야 한다.

v2.0은 새 설치 파일을 만들거나 updater를 추가하더라도 기존 token, chat id, outputs, logs, PCS/PTS 스킬, 모델 인증 구조를 함부로 초기화하지 않는다.

### Web Control Agent

v2.0은 Web Control Agent다.

이 말은 봇이 담당 agent web에 접속해서 조작할 수 있게 된다는 의미다.

v2.2는 Master Builder Agent 단계다.

v2.2는 봇이 스킬 개선안을 만들고, 마스터 승인 후 updater로 반영하는 단계다.

이 두 단계를 섞지 않는다.

### Playwright MCP

Playwright는 브라우저 자동화 엔진이다.

Playwright MCP는 에이전트가 Playwright를 사용할 수 있도록 연결하는 방식이다.

집필 모델은 이 설명을 초보자 눈높이로 반복한다.

### data-testid

`data-testid`는 봇을 위한 이름표다.

디자인이 바뀌어도 봇이 같은 버튼을 찾게 해준다.

### outputs 저장

결과물은 먼저 로컬 outputs에 저장한다.

Telegram 전송은 유저가 원할 때만 한다.

### 승인 정책

영상 생성, 유료 크레딧, 외부 배포, 세무/의료 상담 결과 전달, 프로젝트 삭제, 여러 agent web 동시 실행, 대량 파일 생성/삭제는 마스터 승인이 필요하다.

## 8. 집필 모델이 사용할 수 있는 반복 구조

각 장을 확장할 때 아래 구조를 사용한다.

```text
문제 제기
→ 쉬운 설명
→ Phoenix 시스템에서의 해석
→ 실제 예시
→ 운영 시 주의점
→ 독자가 따라 할 체크리스트
```

이 구조를 쓰면 원고가 길어져도 흐트러지지 않는다.

## 9. 원고 품질 기준

좋은 원고는 세 가지를 동시에 만족해야 한다.

1. 개념이 이해된다.
2. 실습을 따라 할 수 있다.
3. 운영 위험을 피할 수 있다.

단순히 멋진 문장만 있는 원고는 부족하다.

단순히 명령어만 나열한 원고도 부족하다.

이 책은 개념서, 실습서, 운영 가이드의 중간에 있어야 한다.
