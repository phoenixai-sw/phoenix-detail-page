# 입문자용 실습 시나리오

## 1. 이 장의 목표

이 장은 독자가 Phoenix Agent Bots와 Agent Webs 구조를 손으로 따라 해볼 수 있도록 만든 실습 시나리오다.

코드를 완전히 이해하지 못해도 괜찮다.

먼저 흐름을 이해하고, 그 다음 각 도구가 어떤 역할을 하는지 익히면 된다.

## 2. 실습 전체 그림

이번 실습에서는 아래 흐름을 이해한다.

```text
GitHub에서 웹사이트 받기
→ 로컬에서 실행
→ Vercel 또는 Netlify 배포
→ Telegram BotFather로 봇 만들기
→ Phoenix Agent v2.0 설치
→ 봇에게 agent web 준비 상태 확인 요청
→ 결과물을 outputs에 저장
→ 필요하면 Telegram으로 전송
```

입문자는 처음부터 모든 agent web을 만들 필요가 없다.

먼저 `phoenix pages` 하나를 기준으로 연습한 뒤, 같은 원리를 다른 agent web에 적용하면 된다.

## 3. 실습 1: GitHub에서 phoenix pages 받기

사용 저장소:

```text
https://github.com/phoenixai-sw/phoenix-detail-page.git
```

방법 A: GitHub Desktop 사용

1. GitHub Desktop을 연다.
2. `File > Clone repository`를 누른다.
3. URL 입력 칸에 저장소 주소를 넣는다.
4. 로컬 저장 위치를 선택한다.
5. `Clone`을 누른다.
6. 폴더가 생성되었는지 확인한다.

방법 B: ZIP 다운로드

1. GitHub 저장소 페이지에 들어간다.
2. 초록색 `Code` 버튼을 누른다.
3. `Download ZIP`을 누른다.
4. 압축을 푼다.
5. 압축을 푼 폴더를 연다.

성공 기준:

```text
프로젝트 폴더 안에 package.json이 보인다.
```

## 4. 실습 2: 로컬에서 실행하기

터미널을 열고 프로젝트 폴더로 이동한다.

```bash
npm install
npm run dev
```

브라우저에서 아래 주소를 연다.

```text
http://localhost:3000
```

다른 포트가 뜨면 안내된 주소를 그대로 열면 된다.

확인할 것:

1. 메인 화면이 열리는가?
2. 업로드 영역이 보이는가?
3. 생성 옵션이 보이는가?
4. 결과 다운로드 버튼이 보이는가?
5. 오류 메시지가 한글로 이해 가능하게 나오는가?

## 5. 실습 3: Vercel에 phoenix pages 배포하기

`phoenix pages`는 Vercel을 기준으로 설명한다.

절차:

1. Vercel에 로그인한다.
2. `Add New Project`를 누른다.
3. GitHub 저장소를 연결한다.
4. `phoenix-detail-page` 저장소를 선택한다.
5. Framework Preset을 확인한다.
6. 환경변수가 필요하면 입력한다.
7. `Deploy`를 누른다.
8. 배포 URL에 접속한다.

성공 기준:

```text
배포 URL에서 phoenix pages 화면이 열린다.
```

실사용 테스트:

1. 이미지 또는 PDF를 업로드한다.
2. 1장 생성 테스트를 한다.
3. 4장 생성 테스트를 한다.
4. 8장 생성 테스트를 한다.
5. 결과 다운로드를 확인한다.

## 6. 실습 4: Netlify 배포 이해하기

`phoenix pages`를 제외한 나머지 agent web은 Netlify 기준으로 설명한다.

대상 예시:

```text
phoenix images
phoenix dental
phoenix command
phoenix slides
phoenix webs
phoenix books
phoenix videos
phoenix reports
phoenix tax
phoenix marketing
```

Netlify 절차:

1. Netlify에 로그인한다.
2. `Add new site`를 누른다.
3. GitHub 저장소를 연결한다.
4. Build command를 확인한다.
5. Publish directory를 확인한다.
6. 환경변수를 입력한다.
7. Deploy를 누른다.

주의:

Netlify는 정적 웹과 대시보드에 편하다.

장시간 이미지 생성, 영상 생성, 대용량 파일 처리처럼 오래 걸리는 작업은 별도 백엔드와 저장소 구조가 필요할 수 있다.

## 7. 실습 5: Telegram BotFather로 봇 만들기

최종 봇 username 기준:

```text
genesis_bot
power_bot
design_bot
video_bot
writer_bot
```

실제 Telegram에서는 username 중복이 있을 수 있다. 그럴 때는 회사 규칙에 맞춰 접두어나 접미사를 붙인다.

절차:

1. Telegram에서 `@BotFather`를 검색한다.
2. `/newbot`을 입력한다.
3. 봇 이름을 입력한다.
4. 봇 username을 입력한다.
5. 발급된 token을 복사한다.
6. token을 안전한 곳에 보관한다.
7. 원고나 강의자료에는 token을 적지 않는다.

파일 예시:

```text
telegram_access_token.txt
telegram_chat_id.txt
```

## 8. 실습 6: Phoenix Agent v2.0 설치 파일 받기

설치 저장소:

```text
https://github.com/phoenixai-sw/phoenix-agents-camp.git
```

설치 ZIP:

```text
Phoenix Agent_v2.0_win11_260706.zip
Phoenix Agent_v2.0_macOS_260706.zip
```

Windows 11은 `.ps1` 파일을 사용한다.

macOS는 `.command`와 `.sh` 파일을 사용한다.

입문자는 자신의 운영체제에 맞는 ZIP만 사용하면 된다.

## 9. 실습 7: 봇에게 준비 상태 확인 요청하기

설치가 끝나면 각 봇에게 `/start`를 보낸다.

그 다음 아래처럼 요청한다.

Genesis Bot:

```text
phoenix pages를 이용해서 상세페이지 생성 작업을 준비하려고 해.
필요한 봇과 agent web을 구분해서 알려줘.
승인이 필요한 작업도 함께 알려줘.
```

Design Bot:

```text
phoenix pages 작업 화면을 조작할 준비가 되었는지 확인해줘.
업로드, 생성, 다운로드 흐름을 점검하고 결과는 outputs 기준으로 안내해줘.
```

Power Bot:

```text
phoenix marketing처럼 대시보드형 agent web을 운영할 때 필요한 준비물을 알려줘.
```

## 10. 실습 8: 결과물 저장과 Telegram 전송

작업이 끝나면 봇은 먼저 outputs 저장을 안내해야 한다.

권장 문구:

```text
결과물은 로컬 outputs 폴더에 저장했습니다.
텔레그램으로도 결과물을 보실래요?
```

입문자는 여기서 두 가지를 구분해야 한다.

1. 결과물이 생성되었는가?
2. Telegram 전송이 되었는가?

생성은 성공했지만 Telegram 전송만 실패할 수 있다.

이 경우에는 outputs 폴더에서 직접 파일을 확인하면 된다.

## 11. 실습 9: 승인 필요한 작업 구분하기

아래 작업은 자동으로 진행하지 않는다.

1. 영상 생성
2. 유료 크레딧 사용
3. 외부 배포
4. 세무 상담 결과 전달
5. 치과 상담 결과 전달
6. 프로젝트 삭제
7. 여러 agent web 동시 실행
8. 대량 파일 생성/삭제

봇은 아래처럼 물어봐야 한다.

```text
이 작업은 비용이 발생하거나 외부에 공개될 수 있습니다.
진행할까요?

1. 진행
2. 중단
3. 초안만 먼저 작성
```

이 승인 단계는 자동화의 방해물이 아니라 안전장치다.

## 12. 실습 10: 오류가 났을 때 보고 문장 만들기

오류가 났을 때는 “안 됩니다”라고만 말하지 말고 아래 형식으로 정리한다.

```text
어떤 봇:
어떤 agent web:
어떤 작업:
어디까지 성공:
어디서 실패:
화면 메시지:
outputs 파일 생성 여부:
Telegram 전송 여부:
```

예시:

```text
어떤 봇: Design Bot
어떤 agent web: phoenix pages
어떤 작업: 상세페이지 생성
어디까지 성공: 업로드 성공, 생성 시작
어디서 실패: 다운로드 전 타임아웃
화면 메시지: 이미지 생성 응답 시간이 초과되었습니다
outputs 파일 생성 여부: 없음
Telegram 전송 여부: 전송 전
```

이렇게 정리하면 운영자나 개발자가 문제를 훨씬 빠르게 찾을 수 있다.

## 13. 실습 완료 기준

아래 항목을 확인하면 입문자 실습은 완료다.

1. GitHub 저장소에서 웹사이트 파일을 받았다.
2. 로컬에서 실행했다.
3. Vercel 또는 Netlify에 배포했다.
4. Telegram BotFather로 봇을 만들었다.
5. Phoenix Agent v2.0 설치 파일 구조를 이해했다.
6. 봇에게 `/start`를 보내 답장을 받았다.
7. outputs 저장 원칙을 이해했다.
8. Telegram 선택 전송 원칙을 이해했다.
9. 승인 필요한 작업을 구분했다.
10. 오류 보고 문장을 작성할 수 있다.

## 14. 마지막 정리

입문자가 처음부터 완벽한 자동화 시스템을 만들 필요는 없다.

먼저 웹사이트를 열고, 배포하고, 봇을 만들고, 결과물을 저장하는 흐름을 이해하면 된다.

그 다음에 agent web을 늘리고, 봇별 역할을 나누고, `phoenix command`를 통해 전체 작업을 지휘하는 구조로 확장한다.

Phoenix 시스템의 핵심은 한 문장으로 정리할 수 있다.

```text
사람이 직접 쓰는 웹을, 봇도 안정적으로 조작할 수 있게 만드는 것.
```
