# 실전: GitHub에서 웹사이트를 받아 구축하고 배포하기

## 1. 이 장의 목표

이 장은 독자가 회사 GitHub 저장소에서 Phoenix agent web 코드를 받아 로컬에서 실행하고, 배포 플랫폼에 올리는 방법을 다룬다.

여기서 설명하는 웹사이트들은 사람도 직접 사용할 수 있고, Agent Bot v2.0이 컨트롤할 수도 있는 `agent web`이다.

1. `phoenix pages`는 Vercel을 메인 배포 플랫폼으로 설명한다.
2. `phoenix pages`를 제외한 나머지 agent web은 Netlify를 메인 배포 플랫폼으로 설명한다.
3. 예정 저장소는 독자가 이해하기 쉽도록 문서에 먼저 넣어두고, 실제 저장소가 만들어지면 같은 주소로 연결한다.
4. 입문자는 먼저 실제 존재 저장소 3개로 연습하고, 예정 저장소는 구조를 이해하는 용도로 본다.

## 3. GitHub에서 코드 받기

가장 기본 방식은 `git clone`이다.

예시:

```bash
git clone https://github.com/phoenixai-sw/phoenix-detail-page.git
cd phoenix-detail-page
```

초보자는 GitHub Desktop을 사용해도 된다.

절차:

1. GitHub Desktop을 연다.
2. `File > Clone repository`를 누른다.
3. 저장소 URL을 입력한다.
4. 로컬 저장 경로를 선택한다.
5. `Clone`을 누른다.

ZIP 다운로드 방식도 가능하다.

절차:

1. GitHub 저장소 페이지에 접속한다.
2. `Code` 버튼을 누른다.
3. `Download ZIP`을 선택한다.
4. 압축을 푼다.
5. 압축을 푼 폴더에서 터미널을 연다.

## 4. 로컬 실행 준비

대부분의 Phoenix agent web은 Node.js 기반 웹 프로젝트로 구성하는 것을 권장한다.

일반적인 실행 흐름:

```bash
npm install
npm run dev
```

빌드 확인:

```bash
npm run build
```

개발 서버가 켜지면 보통 아래 주소로 접속한다.

```text
http://localhost:3000
```

포트가 이미 사용 중이면 3001, 3002처럼 다른 포트로 실행될 수 있다.

## 5. 환경변수 설정

환경변수는 `.env.local` 또는 배포 플랫폼의 Environment Variables 메뉴에 넣는다.

예시:

```text
OPENAI_API_KEY=여기에_키_입력
GOOGLE_API_KEY=여기에_키_입력
SUPABASE_URL=여기에_URL_입력
SUPABASE_ANON_KEY=여기에_ANON_KEY_입력
```

주의:

1. 실제 API 키는 GitHub에 올리지 않는다.
2. `.env.local`은 `.gitignore`에 포함되어야 한다.
3. 출판 원고와 강의자료에는 실제 키를 쓰지 않는다.
4. 수강생에게는 “어디에 넣는지”만 알려주고, 실제 값은 각자 발급하게 한다.

## 6. Vercel 배포

Vercel은 `phoenix pages`의 메인 배포 플랫폼이다. 상세페이지 생성처럼 이미지 생성 시간이 길어질 수 있는 웹서비스는 Vercel 기준으로 먼저 설명한다.

절차:

1. Vercel에 로그인한다.
2. `Add New Project`를 누른다.
3. GitHub 저장소를 선택한다.
4. Framework Preset을 확인한다.
5. Environment Variables를 입력한다.
6. `Deploy`를 누른다.

배포 후 확인:

```text
Production URL 접속
생성 기능 테스트
다운로드 기능 테스트
환경변수 누락 여부 확인
```

입문자 확인 포인트:

1. Vercel 프로젝트 이름이 `phoenix pages` 또는 `phoenix-detail-page`와 연결되어 있는지 확인한다.
2. GitHub 저장소가 `https://github.com/phoenixai-sw/phoenix-detail-page.git`인지 확인한다.
3. 이미지 생성에 필요한 API 키를 넣는 방식이 프로젝트 정책과 맞는지 확인한다.
4. 배포 후 실제로 1장, 4장, 8장 생성 테스트를 해본다.

## 7. Netlify 배포

Netlify는 `phoenix pages`를 제외한 나머지 agent web의 메인 배포 플랫폼이다.

절차:

1. Netlify에 로그인한다.
2. `Add new site`를 누른다.
3. GitHub 저장소를 연결한다.
4. Build command를 설정한다.
5. Publish directory를 설정한다.
6. Environment variables를 입력한다.
7. Deploy를 실행한다.

Netlify 기준 배포 대상:

1. `phoenix command`
2. `phoenix images`
3. `phoenix slides`
4. `phoenix webs`
5. `phoenix books`
6. `phoenix videos`
7. `phoenix reports`
8. `phoenix tax`
9. `phoenix dental`
10. `phoenix marketing`

주의:

1. 이미지 생성처럼 응답 시간이 긴 기능은 Netlify에서 타임아웃에 더 민감할 수 있다.
2. 장시간 생성 기능이 필요한 경우 백엔드 구조, 비동기 작업, 저장소 연동을 별도로 설계해야 한다.
3. 단순 랜딩페이지, 대시보드, 결과 조회, 문서형 웹은 Netlify로 운영하기 좋다.

## 8. Agent Bot이 컨트롤하기 좋은 웹의 조건

Agent Bot v2.0이 agent web을 안정적으로 컨트롤하려면 웹사이트가 아래 조건을 갖춰야 한다.

1. 주요 버튼과 입력창에 `data-testid`가 있다.
2. 업로드 흐름이 안정적이다.
3. 생성 상태가 화면에 표시된다.
4. 실패 상태가 명확하게 표시된다.
5. 결과 다운로드 버튼이 안정적이다.
6. 결과물이 로컬 `outputs` 또는 Cloud에 저장된다.
7. API key/token이 화면이나 로그에 노출되지 않는다.

예시:

```tsx
<button data-testid="phoenix-images-generate">
  이미지 생성
</button>
```

```tsx
<a data-testid="phoenix-pages-download-zip" href={zipUrl}>
  ZIP 다운로드
</a>
```

## 9. 운영 체크리스트

배포 후 아래 항목을 확인한다.

1. 로컬 `npm run build` 통과
2. 배포 URL 접속 성공
3. 환경변수 입력 완료
4. 업로드 기능 정상
5. 생성 기능 정상
6. 다운로드 기능 정상
7. 모바일 화면 확인
8. 실패 메시지 확인
9. Agent Bot 자동화에 필요한 `data-testid` 확인
10. GitHub 저장소에 비밀키가 올라가지 않았는지 확인

## 10. 입문자 실습: phoenix pages를 Vercel에 올려보기

이 실습은 독자가 가장 먼저 따라 할 수 있는 흐름이다.

목표:

```text
phoenix pages 저장소를 받아서 로컬에서 실행하고, Vercel에 배포한다.
```

사용 저장소:

```text
https://github.com/phoenixai-sw/phoenix-detail-page.git
```

단계:

1. GitHub 저장소 페이지에 접속한다.
2. GitHub Desktop으로 clone하거나 ZIP으로 다운로드한다.
3. 압축을 풀거나 clone된 폴더를 연다.
4. 터미널에서 프로젝트 폴더로 이동한다.
5. `npm install`을 실행한다.
6. `npm run dev`를 실행한다.
7. 브라우저에서 `http://localhost:3000` 또는 안내된 localhost 주소를 연다.
8. 화면이 열리면 업로드, 생성, 다운로드 버튼이 보이는지 확인한다.
9. `npm run build`를 실행해 빌드가 통과하는지 확인한다.
10. Vercel에서 GitHub 저장소를 연결하고 배포한다.

초보자가 자주 막히는 지점:

1. Node.js가 설치되어 있지 않다.
2. 터미널 위치가 프로젝트 폴더가 아니다.
3. `.env.local`이 필요한데 값이 없다.
4. 3000번 포트가 이미 사용 중이다.
5. Vercel에 GitHub 권한을 주지 않았다.
6. 환경변수를 로컬에는 넣었지만 Vercel에는 넣지 않았다.

확인 문장:

```text
로컬에서 열림
빌드 통과
Vercel 배포 URL 접속
생성 기능 테스트
다운로드 테스트
```

이 다섯 가지가 확인되면 첫 배포 실습은 성공이다.

## 11. Netlify 실습 기준

Netlify는 `phoenix pages`를 제외한 나머지 agent web의 기본 배포 기준이다.

입문자는 처음부터 모든 웹을 만들 필요가 없다. 먼저 아래 순서로 이해하면 된다.

1. `phoenix images`처럼 실제 저장소가 있는 웹을 Netlify에 연결한다.
2. Build command와 Publish directory를 확인한다.
3. Environment variables를 넣는다.
4. 배포 URL에 접속한다.
5. 생성형 기능이 있다면 응답 시간이 긴 작업에서 타임아웃이 없는지 확인한다.

주의:

Netlify는 정적 웹, 랜딩페이지, 대시보드, 문서형 웹에 특히 편하다.

장시간 이미지 생성, 영상 생성, 대용량 파일 처리처럼 시간이 오래 걸리는 작업은 별도 백엔드, 비동기 작업, 저장소 연동을 고려해야 한다.

## 12. Agent Bot이 조작하기 좋은 웹 점검표

웹사이트를 사람만 쓰는 경우와 봇도 쓰는 경우는 점검 기준이 다르다.

아래 항목이 준비되어야 Agent Bot이 안정적으로 조작할 수 있다.

1. 주요 입력창에 `data-testid`가 있다.
2. 주요 버튼에 `data-testid`가 있다.
3. 업로드 성공/실패 상태가 화면에 표시된다.
4. 생성 중 상태가 화면에 표시된다.
5. 실패 상태와 재시도 방법이 표시된다.
6. 결과 미리보기가 있다.
7. 결과 다운로드 버튼이 있다.
8. 다운로드 결과 파일명이 예측 가능하다.
9. 로그인 만료 시 안내가 나온다.
10. 민감값이 화면이나 로그에 노출되지 않는다.

예시:

```tsx
<input data-testid="phoenix-pages-upload-input" type="file" />
<button data-testid="phoenix-pages-generate">생성하기</button>
<button data-testid="phoenix-pages-download-zip">ZIP 다운로드</button>
```

이런 이름표가 많아질수록 봇은 화면 디자인 변화에 덜 흔들린다.
