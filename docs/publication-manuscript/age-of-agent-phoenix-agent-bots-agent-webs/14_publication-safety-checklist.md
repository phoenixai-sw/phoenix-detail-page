# 출판 편집 안전 검수 체크리스트

## 1. 이 문서의 목적

이 문서는 출판 전 원고를 검수하기 위한 체크리스트다.

GPT 5.5, Claude Opus 4.8, Fable이 원고를 확장하더라도 마지막에는 이 기준으로 다시 확인해야 한다.

## 2. 기술 정확성 체크

확인 항목:

1. `phoenix pages`만 Vercel 기준으로 설명했는가?
2. 나머지 agent web은 Netlify 기준으로 설명했는가?
3. GitHub 저장소 URL이 정확한가?
4. 실제 존재 저장소와 예정 저장소를 구분했는가?
5. Phoenix Agent v2.0 설치 저장소를 정확히 적었는가?
6. Windows 11과 macOS 설치 파일명을 구분했는가?
7. Windows는 `.ps1`, macOS는 `.command`와 `.sh` 기준으로 설명했는가?
8. v2.0을 Web Control Agent로 설명했는가?
9. v2.2를 Master Builder Agent로 구분했는가?
10. v1.9 안정 기능 보존 원칙을 빠뜨리지 않았는가?

## 3. 보안 체크

원고에 아래 값이 실제 값으로 들어가면 안 된다.

1. OpenAI API key
2. Google API key
3. Telegram token
4. Telegram chat id
5. OAuth credential
6. Supabase service role key
7. DATABASE_URL 실제값
8. `.env` 실제값
9. 개인 이메일 비밀번호
10. 배포 플랫폼 접근 token

권장 표현:

```text
여기에_키_입력
YOUR_TOKEN_HERE
실제 값은 원고에 적지 않는다.
```

## 4. 승인 정책 체크

아래 작업은 반드시 승인 필요로 설명한다.

1. 영상 생성
2. 유료 크레딧 사용
3. 외부 공개 배포
4. 세무 상담 결과 전달
5. 치과 상담 결과 전달
6. 프로젝트 삭제
7. 여러 agent web 동시 실행
8. 대량 파일 생성
9. 대량 파일 삭제
10. 운영 중인 로컬 활성 봇 수정

승인 문구 예시:

```text
이 작업은 비용이 발생하거나 외부에 공개될 수 있습니다.
진행할까요?

1. 진행
2. 중단
3. 초안만 먼저 작성
```

## 5. 세무/의료 안전 표현 체크

`phoenix tax`와 `phoenix dental` 관련 원고는 아래 기준을 지킨다.

1. 일반 정보라고 밝힌다.
2. 전문가 확인이 필요하다고 밝힌다.
3. AI가 최종 판단을 대신한다고 쓰지 않는다.
4. 진단, 처방, 세액 확정처럼 오해될 표현을 피한다.
5. 상담 결과 전달은 마스터 승인 후 진행한다고 쓴다.

권장 문구:

```text
이 결과는 일반 정보이며, 실제 세무 판단은 세무사 확인이 필요합니다.
```

```text
이 결과는 일반적인 치아관리 정보이며, 진단과 치료 결정은 치과 전문의 상담이 필요합니다.
```

## 6. 배포 플랫폼 표현 체크

피해야 할 표현:

1. Vercel이면 모든 생성 작업이 무조건 안정적이다.
2. Netlify에서는 생성형 기능을 절대 쓸 수 없다.
3. 환경변수를 한 번 넣으면 어떤 환경에서도 자동 적용된다.
4. 배포 후 테스트하지 않아도 된다.
5. 플랫폼 요금과 제한은 고정되어 있다.

권장 표현:

```text
배포 플랫폼의 정책과 제한은 바뀔 수 있으므로 실제 운영 전 공식 문서를 확인한다.
```

```text
장시간 생성 작업은 배포 플랫폼, 백엔드 구조, 저장소 방식에 따라 안정성이 달라진다.
```

## 7. Agent Web 자동화 체크

원고에 아래 개념이 들어가야 한다.

1. Playwright는 브라우저 자동화 엔진이다.
2. Playwright MCP는 에이전트가 Playwright를 쓰게 하는 연결 방식이다.
3. `data-testid`는 봇을 위한 이름표다.
4. 생성 상태와 실패 상태가 화면에 표시되어야 한다.
5. 결과 다운로드 버튼은 안정적으로 찾아야 한다.
6. 로그인 만료 시 재로그인 안내가 필요하다.
7. 민감값은 화면과 로그에 노출하지 않는다.

## 8. outputs와 Telegram 체크

원고는 아래 기준을 유지한다.

1. 결과물은 먼저 outputs에 저장한다.
2. Telegram 전송은 유저에게 물어본 뒤 진행한다.
3. 파일이 크면 ZIP 또는 Cloud 링크를 고려한다.
4. 생성 실패와 전송 실패를 구분한다.
5. 결과물이 없는데 완료라고 말하지 않는다.
