# V1 실행을 위해 코덱스에 입력할 작업지시서

이 문서는 V0 상태의 `phoenix pages` GitHub 저장소를 내려받은 수강생이 코덱스에게 그대로 붙여넣어 V1 업그레이드 작업을 요청하기 위한 작업지시서이다.

V1의 목표는 `브라우저 저장 + 로컬 다운로드` 중심의 V0를 `Supabase Auth + Supabase Database + Supabase Storage` 기반의 클라우드 저장형 서비스로 업그레이드하는 것이다.

---

## 1. 코덱스에게 붙여넣을 시작 문장

아래 내용을 코덱스 채팅창에 그대로 붙여넣는다.

```text
현재 저장소는 phoenix pages V0 상태입니다.
이 프로젝트를 V1으로 업그레이드해주세요.

V1 목표는 다음과 같습니다.

1. Supabase Auth 기반 Google 로그인 추가
2. Supabase Storage에 생성 이미지 저장
3. Supabase Database에 프로젝트와 섹션 정보 저장
4. 유저별 작업 목록과 다시 열기 기능 추가
5. 슈퍼 관리자와 일반 유저 관리자 구분
6. 기존 V0 브라우저 저장과 로컬 다운로드 기능은 fallback으로 유지
7. 이미지 데이터가 너무 커서 편집 요청이 실패하는 문제를 Storage URL 기반 구조로 완화
8. Vercel 배포 기준으로 환경변수와 빌드 검증까지 확인

코드 수정 전 현재 구조를 분석하고,
수정할 파일 목록과 구현 순서를 먼저 짧게 보고한 뒤 작업을 진행해주세요.
작업 완료 후 변경 파일, 환경변수 목록, Supabase SQL, 테스트 방법을 보고해주세요.
```

## 2. 코덱스에게 제공할 준비물

마스터가 준비한 값 중 공개 가능한 값과 비밀 값을 구분해서 제공한다.

비밀 값은 채팅창에 직접 붙여넣지 않고, 가능하면 `.env.local` 또는 Vercel 환경변수에 마스터가 직접 입력한다.

### 2.1 공개 가능 값

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=
SUPABASE_STORAGE_BUCKET=phoenix-pages-images
SUPER_ADMIN_EMAILS=
```

`NEXT_PUBLIC_`로 시작하는 값은 브라우저에 노출될 수 있는 값이다.

### 2.2 비밀 값

아래 값은 채팅창, GitHub, 스크린샷, 문서에 공개하지 않는다.

```env
SUPABASE_SERVICE_ROLE_KEY=
SUPABASE_SECRET_KEY=
GOOGLE_CLIENT_SECRET=
DATABASE_URL=
```

코덱스에게는 이렇게 말한다.

```text
비밀 환경변수는 제가 Vercel과 로컬 .env.local에 직접 입력하겠습니다.
코드에서는 환경변수 이름만 사용해주세요.
```

## 3. V1 구현 범위

코덱스는 아래 범위를 구현한다.

### 3.1 Supabase 연결

1. Supabase client 설정 파일을 만든다.
2. 브라우저용 client와 서버용 admin client를 분리한다.
3. 공개용 key와 secret key를 섞지 않는다.
4. `NEXT_PUBLIC_` 값은 client에서만 사용한다.
5. service role 또는 secret key는 server route에서만 사용한다.

권장 파일 예시:

```text
src/lib/supabase/client.ts
src/lib/supabase/server.ts
src/lib/supabase/admin.ts
```

### 3.2 Google 로그인

1. Supabase Auth의 Google Provider를 사용한다.
2. 로그인 버튼을 추가한다.
3. 로그아웃 버튼을 추가한다.
4. 로그인한 유저의 이메일과 이름을 표시한다.
5. 로그인하지 않아도 V0 방식으로 체험은 가능하게 둔다.
6. 로그인한 유저는 클라우드 저장을 사용할 수 있게 한다.

### 3.3 Database 테이블

코덱스는 Supabase SQL Editor에 넣을 SQL을 작성해야 한다.

필수 테이블:

```text
profiles
projects
project_sections
admin_users
```

권장 역할:

| 테이블 | 역할 |
|---|---|
| `profiles` | 로그인 유저 기본 정보 |
| `projects` | 프로젝트 제목, 유저, 채널, 모델, 생성 상태 |
| `project_sections` | 섹션별 이미지 URL, 프롬프트, 편집 이력 |
| `admin_users` | 슈퍼 관리자 권한 |

### 3.4 RLS 정책

Row Level Security는 반드시 켠다.

기본 원칙:

1. 일반 유저는 자기 프로젝트만 볼 수 있다.
2. 일반 유저는 자기 프로젝트만 수정할 수 있다.
3. 슈퍼 관리자는 전체 프로젝트를 볼 수 있다.
4. service role key는 서버 작업에만 사용한다.
5. Storage 파일도 유저별 경로로 분리한다.

Storage 경로 예시:

```text
users/{user_id}/projects/{project_id}/sections/{section_id}.jpg
```

### 3.5 Storage 저장

생성 이미지 저장 구조를 아래처럼 바꾼다.

V0:

```text
AI 생성 이미지
→ base64 data URL
→ 브라우저 상태와 localStorage에 저장
```

V1:

```text
AI 생성 이미지
→ 서버에서 Supabase Storage 업로드
→ Storage path 또는 signed/public URL 반환
→ Database에 이미지 URL과 path 저장
→ 화면은 URL로 이미지 표시
```

필수 구현:

1. 이미지 upload API
2. Storage path 생성 규칙
3. 이미지 URL 반환
4. DB에 프로젝트와 섹션 저장
5. 기존 base64 방식 fallback 유지

### 3.6 프로젝트 저장

로그인한 유저가 `결과 저장`을 누르면 클라우드에 저장한다.

저장 항목:

```text
project id
owner user id
title
channel
ratio
model
model label
request text
analysis summary
status
created at
updated at
sections
image storage path
image URL
prompt
edit history
```

### 3.7 작업 목록

현재 V0의 최근 제작 작업은 브라우저 저장 기반이다.

V1에서는 로그인 상태에 따라 다르게 동작한다.

```text
비로그인 유저
→ 브라우저 최근 작업 표시

로그인 일반 유저
→ 자기 Supabase projects 목록 표시

슈퍼 관리자
→ 전체 유저 projects 목록 표시
```

### 3.8 슈퍼 관리자 모드

슈퍼 관리자 이메일은 환경변수 또는 `admin_users` 테이블 기준으로 판단한다.

슈퍼 관리자 기능:

1. 전체 유저 프로젝트 보기
2. 유저 이메일 기준 검색
3. 프로젝트 상태 확인
4. 이미지 저장 경로 확인
5. 오류 작업 확인

처음 V1에서는 삭제와 강제 수정 기능은 최소화한다.
조회 중심으로 먼저 만든다.

### 3.9 V0 fallback 유지

V1 작업 중에도 기존 기능은 깨지면 안 된다.

반드시 유지할 기능:

1. API 키 설정
2. 이미지/PDF 업로드
3. 1장, 4장, 8장 생성
4. 9:16, 1:1, 4:5 출력 비율
5. 결과 다운로드 ZIP
6. 개별 편집
7. AI 멘트 생성
8. 페이지 추가 생성
9. 브라우저 저장 기반 최근 작업

로그인과 Supabase 설정이 없을 때도 V0 방식으로 사용할 수 있어야 한다.

## 4. 코덱스에게 요구할 작업 순서

코덱스는 아래 순서로 진행한다.

### 4.1 현재 코드 분석

1. `package.json` 확인
2. `src/components/redesign-wizard.tsx` 확인
3. `/api/generate` 확인
4. `/api/edit-section` 확인
5. 현재 저장 로직 확인
6. 현재 이미지 데이터 처리 방식 확인

### 4.2 의존성 추가

필요 시 Supabase 패키지를 추가한다.

예:

```bash
npm install @supabase/supabase-js
```

이미 설치되어 있으면 중복 설치하지 않는다.

### 4.3 환경변수 연결

`.env.example` 또는 문서에 필요한 환경변수를 정리한다.

필수 후보:

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=
SUPABASE_SERVICE_ROLE_KEY=
SUPABASE_SECRET_KEY=
SUPABASE_STORAGE_BUCKET=phoenix-pages-images
SUPER_ADMIN_EMAILS=
```

### 4.4 Supabase SQL 작성

코덱스는 SQL 파일 또는 MD 문서로 SQL을 제공한다.

권장 경로:

```text
docs/upgrade-roadmap/v1-phoenix-cloud-save/v1_supabase_schema.sql
```

또는:

```text
supabase/schema.sql
```

### 4.5 API 구현

필요한 API 예시:

```text
GET /api/projects
POST /api/projects
GET /api/projects/[id]
PATCH /api/projects/[id]
POST /api/storage/upload-image
GET /api/admin/projects
```

프로젝트 구조에 맞게 이름은 조정해도 된다.

### 4.6 UI 구현

추가 UI:

1. 로그인 버튼
2. 로그아웃 버튼
3. 유저 정보 표시
4. 클라우드 저장 상태 표시
5. 슈퍼 관리자 모드 진입 표시
6. 클라우드 작업 목록
7. 저장 실패 시 V0 fallback 안내

### 4.7 테스트

최소 검증:

```bash
npm run build
```

가능하면 로컬 실행:

```bash
npm run dev:local
```

브라우저 확인:

```text
http://localhost:3002
```

작업이 끝나면 코덱스는 아래 형식으로 보고한다.

```text
V1 작업 완료 보고

1. 변경 파일
- 파일 경로 1
- 파일 경로 2

2. 추가된 환경변수
- NEXT_PUBLIC_SUPABASE_URL
- ...

3. Supabase에서 실행해야 할 SQL
- 경로 또는 코드 블록

4. 마스터가 해야 할 배포 설정
- Vercel 환경변수
- Supabase Auth URL
- Google OAuth Redirect URI

5. 테스트 결과
- npm run build 성공/실패
- 로컬 실행 확인 여부

6. 남은 주의사항
- V1.5 결제 전까지 크레딧 기능 없음
- V2 봇 자동화 전까지 수동 웹 사용 중심
```

아래 사항은 하지 않도록 지시한다.

## 8. MCP 사용 가능 시 추가 지시

수강생 환경에 Supabase MCP가 연결되어 있다면 코덱스에게 아래 지시를 추가한다.

```text
가능하면 Supabase MCP를 사용해서 프로젝트 상태를 확인해주세요.

MCP로 확인할 것:
1. Supabase 프로젝트 연결 상태
2. Project URL
3. 공개용 key 종류
4. Storage bucket 존재 여부
5. 기존 테이블 목록
6. RLS 활성화 여부
7. SQL 실행 가능 여부
8. 오류 로그

처음에는 read-only 모드로 확인하고,
SQL이나 migration 적용이 필요할 때만 제 확인을 받고 쓰기 작업을 진행해주세요.
```

Supabase MCP 공식 endpoint:

```text
https://mcp.supabase.com/mcp
```

프로젝트 제한 권장 형식:

```text
https://mcp.supabase.com/mcp?project_ref=YOUR_PROJECT_REF
```

초기 확인용 권장 형식:

```text
https://mcp.supabase.com/mcp?project_ref=YOUR_PROJECT_REF&read_only=true&features=database,docs,storage
```

주의:

```text
수강생의 Supabase 프로젝트에만 연결하세요.
마스터 프로젝트나 다른 수강생 프로젝트에 연결하지 마세요.
service role key나 secret key를 브라우저 코드에 넣지 마세요.
```

## 9. MCP를 사용할 때 코덱스 작업 순서

MCP가 연결된 경우 코덱스는 아래 순서로 진행한다.

### 9.1 읽기 전용 확인

1. Supabase MCP 연결 여부 확인
2. project_ref가 현재 수강생 프로젝트인지 확인
3. 프로젝트 URL 확인
4. 기존 table 목록 확인
5. Storage bucket 목록 확인
6. RLS 상태 확인
7. 현재 Auth 설정은 마스터에게 확인 요청

### 9.2 설계 보고

쓰기 작업 전 코덱스는 아래를 보고한다.

```text
확인한 Supabase 프로젝트:
확인한 Storage bucket:
생성할 테이블:
적용할 RLS 정책:
수정할 코드 파일:
마스터가 직접 해야 할 Google OAuth 설정:
```

### 9.3 승인 후 쓰기 작업

마스터가 승인하면 아래 작업을 진행한다.

1. SQL migration 실행 또는 SQL 파일 생성
2. Storage bucket 없으면 생성 SQL/안내 제공
3. RLS 정책 적용
4. 코드에 Supabase 연결
5. 프로젝트 저장 API 구현
6. 이미지 Storage 업로드 구현
7. 빌드 검증

### 9.4 Google Provider는 수동 확인

코덱스는 Google OAuth Client Secret을 요구하지 않는다.

마스터에게 이렇게 안내한다.

```text
Google OAuth Client ID와 Secret은 마스터가 Google Cloud Console에서 직접 생성해주세요.
Supabase Dashboard > Authentication > Providers > Google에 직접 입력해주세요.
저는 코드에서 Supabase Auth Google login을 호출하도록 구현하겠습니다.
```

## 10. MCP 포함 최종 붙여넣기 프롬프트

아래는 MCP 사용 가능 환경에서 코덱스에게 붙여넣을 최종 프롬프트다.

