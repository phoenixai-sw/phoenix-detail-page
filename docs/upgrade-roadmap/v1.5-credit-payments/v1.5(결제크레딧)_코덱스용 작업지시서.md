# V1.5(결제크레딧) 코덱스용 작업지시서

## 1. 작업 목표

V1.5는 V1의 Google 로그인, Supabase DB, Supabase Storage, 슈퍼 관리자/일반 유저 구조 위에 결제와 크레딧 관리를 붙이는 단계다.

목표:

1. Toss 결제 연동
2. 결제 금액만큼 크레딧 지급
3. 신규 유저 무료 크레딧 소량 제공
4. 이미지 생성/편집 시 크레딧 차감
5. 슈퍼 관리자 크레딧/결제 관리
6. 일반 유저 내 크레딧 확인

## 2. 실행 가능 여부

실행 가능하다.

단, V1.5는 V1 없이 바로 진행하면 안 된다. 결제와 크레딧은 유저 ID, 결제 이력, 사용량 이력, 환불 이력이 모두 필요하므로 Supabase Auth와 Database가 먼저 안정화되어야 한다.

## 3. 전제 조건

1. V1 Google 로그인 완료
2. V1 Supabase Database 완료
3. V1 Supabase Storage 완료
4. `profiles`, `user_roles`, `cloud_projects` 테이블 존재
5. 슈퍼 관리자 화면 존재
6. Toss 결제 테스트 키 또는 실서비스 키 확보
7. 결제 성공/실패/취소 redirect URL 확정
8. 결제 webhook 수신 URL 확정

## 3.1 Toss 공식 연동 원칙

Toss 결제는 클라이언트 결제창 성공 화면만으로 완료 처리하지 않는다.

필수 원칙:

1. 브라우저에서는 결제창 또는 결제위젯을 띄운다.
2. 결제 인증 후 서버에서 secret key로 결제 승인 confirm을 수행한다.
3. 결제 상태 변경은 webhook으로도 수신한다.
4. confirm과 webhook은 중복 호출될 수 있으므로 `order_id` 또는 `payment_key` 기준으로 1회만 크레딧을 지급한다.
5. 클라이언트가 보낸 금액, 상태, 성공 여부는 서버에서 다시 검증한다.

## 4. 권장 테이블

### 4.1 credit_accounts

유저별 현재 크레딧 잔액을 빠르게 조회하는 테이블이다.

```text
user_id uuid primary key references auth.users(id)
balance integer not null default 0
free_balance integer not null default 0
paid_balance integer not null default 0
updated_at timestamptz
```

### 4.2 credit_ledger

크레딧의 모든 변화를 기록하는 원장 테이블이다.

```text
id uuid primary key
user_id uuid references auth.users(id)
type text
amount integer
balance_after integer
source text
source_id text
memo text
created_at timestamptz
```

`type` 예시:

```text
signup_bonus
payment_charge
image_generation
image_edit
manual_adjustment
refund
expire
```

### 4.3 payment_orders

결제 주문과 승인 상태를 저장한다.

```text
id uuid primary key
user_id uuid references auth.users(id)
order_id text unique
payment_key text
amount integer
credit_amount integer
status text
provider text default 'toss'
requested_at timestamptz
approved_at timestamptz
cancelled_at timestamptz
raw_payload jsonb
```

### 4.4 usage_logs

이미지 생성/편집 사용량을 저장한다.

```text
id uuid primary key
user_id uuid references auth.users(id)
project_id uuid
action text
engine text
page_count integer
credit_cost integer
status text
created_at timestamptz
```

## 6. API 구현 범위

추가 API 예시:

```text
src/app/api/billing/products/route.ts
src/app/api/billing/orders/route.ts
src/app/api/billing/confirm/route.ts

```

기능:

1. 결제 상품 목록 제공
2. 결제 주문 생성
3. Toss 결제 승인
4. Toss webhook 수신
5. 크레딧 지급
6. 크레딧 차감
7. 크레딧 잔액 조회
8. 슈퍼 관리자 수동 지급/차감

## 7. 크레딧 차감 원칙

1. 생성 시작 전 예상 크레딧을 확인한다.
2. 잔액이 부족하면 생성 버튼을 비활성화한다.
3. 실제 차감은 작업 성공 시점에 한다.
4. 실패한 생성은 차감하지 않는다.
5. 중복 요청을 막기 위해 `usage_logs`와 `credit_ledger`에 idempotency key를 둔다.
6. 같은 결제 승인 webhook이 여러 번 와도 크레딧은 한 번만 지급한다.

## 8. 무료 크레딧 정책

신규 가입 시 무료 크레딧을 1회 지급한다.

권장:

```text
가입 무료 크레딧 = 5~10 크레딧
```

주의:

1. 같은 Supabase user_id에는 한 번만 지급한다.
2. 무료 크레딧과 유료 크레딧을 구분한다.
3. 차감 순서는 무료 크레딧 먼저, 유료 크레딧 나중을 기본값으로 둔다.

## 9. UI 구현 범위

일반 유저 화면:

1. 내 크레딧 잔액
2. 무료 크레딧/유료 크레딧 구분
3. 크레딧 충전 버튼
4. 결제 상품 선택
5. 결제 완료 안내
6. 사용 내역

슈퍼 관리자 화면:

1. 유저별 크레딧 잔액
2. 결제 내역
3. 사용량 내역
4. 수동 크레딧 지급
5. 수동 크레딧 차감
6. 환불/취소 상태 확인

## 10. 금지 사항

1. 결제 성공 redirect만 믿고 크레딧을 지급하지 않는다.
2. webhook 중복 수신 시 크레딧을 중복 지급하지 않는다.
3. 결제 실패 작업에 크레딧을 지급하지 않는다.
4. 이미지 생성 실패에 크레딧을 차감하지 않는다.
5. Toss secret key를 브라우저에 노출하지 않는다.
6. 슈퍼 관리자 권한 없이 다른 유저 크레딧을 수정하지 못하게 한다.

## 11. 검증 기준

1. 신규 유저 가입 시 무료 크레딧 1회 지급
2. 결제 성공 시 크레딧 지급
3. 결제 실패 시 크레딧 미지급
4. webhook 중복 수신 시 1회만 지급
5. 이미지 생성 성공 시 크레딧 차감
6. 이미지 생성 실패 시 크레딧 미차감
7. 잔액 부족 시 생성 차단
8. 슈퍼 관리자에서 유저별 결제/크레딧 확인 가능
9. 일반 유저는 자기 결제/크레딧만 확인 가능
